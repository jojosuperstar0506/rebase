## 3. The three critical issues

### 🔴 Issue 1 — Buggy zero-follower scrapes are poisoning the scoring

**What I found:** Songmont has **5 scrape rows on 2026-04-21 with `followers=0`** plus 1 good row with `followers=321000`. These zeros came from XHS auth-walled responses BEFORE my scraper hardening landed. They're still in the DB.

**The scoring pipeline reads them as legitimate snapshots** and computes:

```
voice_volume scoring inputs (Songmont, latest):
  growth_rate:        -70.0     ← because zeros polluted the trend
  follower_growth:   -100.0     ← because most recent rows show 0
  voice_share_pct:      0.0     ← zero share
  engagement_growth: -100.0
  → score: 15  (low)
```

**Why this is wrong:** Songmont actually has **321K XHS + 434K Douyin = ~755K total followers**. CASSILE has ~80K. By any honest measure, Songmont's voice volume should be ~5× higher than CASSILE — but the buggy zeros make it score 15 vs CASSILE's 85.

**Downstream impact:** the brief verdict reads:
> "古良吉吉强势反弹，Songmont三大维度齐跌需警惕"
> (古良吉吉 is rebounding strongly; Songmont's 3 dimensions are dropping)

This is **factually wrong** — the "drop" is an artifact of bad scrape data, not a real movement. A customer reading this brief would make the wrong strategic decision (e.g., "fix our XHS engagement" when XHS engagement is actually fine; we just had a failed scrape).

**Fix (~5 min):**
```sql
DELETE FROM scraped_brand_profiles
 WHERE follower_count = 0
   AND scraped_at::date = '2026-04-21'
   AND brand_name = 'Songmont';
```
Then re-run scoring + brief generation. Zero impact on real signal — these rows had no useful data anyway.

**Architectural fix (separate ticket):** scoring pipelines should treat `follower_count=0` as `no_data`, not `legitimate_zero`. Currently they're conflated. This is a one-line guard in each scoring pipeline:
```python
if profile.follower_count == 0 and profile.engagement_metrics.get('total_likes', 0) == 0:
    continue  # not a real snapshot
```

### 🟠 Issue 2 — Duplicate scoring rows (5× per metric × brand × day)

**What I found:** Most (metric_type, brand, day) tuples have **5 duplicate rows** on 2026-04-30. The frontend probably picks the "latest" (whichever has the latest timestamp) but it's wasteful and looks like the pipeline ran 5 times.

```
consumer_domain × CASSILE × 2026-04-30 → 5 rows
product_domain  × CASSILE × 2026-04-30 → 5 rows
marketing_domain × 古良吉吉 × 2026-04-30 → 5 rows
... (same for all 3 brands × 3 domains)
```

**Why this happened:** Will's `run_analysis_for_workspace.sh` orchestrator likely loops over each domain pipeline, and the domain aggregation runs after EACH sub-metric finishes — re-aggregating partial results 5 times instead of once at the end.

**Fix:** modify `domain_aggregation_pipeline.py` to either:
- Use `INSERT ... ON CONFLICT (workspace_id, competitor_name, metric_type, metric_version) DO UPDATE` (UPSERT)
- OR delete prior-run rows for the same (workspace, metric, brand, day) before insert

**Also a cleanup query** worth running today:
```sql
-- Keep only the latest analysis_results row per (workspace, metric, brand, version, day)
WITH ranked AS (
  SELECT id, ROW_NUMBER() OVER (
    PARTITION BY workspace_id, metric_type, competitor_name, metric_version, analyzed_at::date
    ORDER BY analyzed_at DESC
  ) AS rn
  FROM analysis_results
)
DELETE FROM analysis_results WHERE id IN (SELECT id FROM ranked WHERE rn > 1);
```

**Severity:** medium — the UI works, but the table grows 5× faster than it should and cron runtimes are 5× slower than necessary.

### 🟠 Issue 3 — LLM hallucinated deltas in the brief moves

**What I found:** The brief move "Songmont心智-13" (mindshare down 13) doesn't reconcile with the underlying scores:
- `consumer_mindshare` Songmont: previous=60, latest=35 → actual delta is **-25, not -13**
- `product_domain` Songmont: latest=72; brief says "from 82 to 72 (-10)" but **82 doesn't appear anywhere in the analysis_results history**

**Why this happens:** DeepSeek is given the current scores + previous scores, and asked to narrate. Sometimes it computes the delta correctly; sometimes it invents a plausible-sounding but wrong number. Will's coercer pass drops moves that reference UNKNOWN brands but doesn't validate that NUMBERS in the moves match the source data.

**Fix:** add a numeric-coherence check to the brand_positioning coercer:
```python
# In brand_positioning_pipeline.py, after parse:
for move in moves:
    # Find any "+N" / "-N" / "to N" pattern in headline + detail + so_what
    for match in re.finditer(r'[+\-]?\d+', move.detail + ' ' + move.headline):
        n = int(match.group())
        # Cross-check against actual score deltas in the inputs
        if not _is_supported_by_inputs(n, brand=move.brand, inputs=raw_metric_inputs):
            logger.warning(f"Move {move.id} cites unsupported number {n}; flagging for review")
            move.detail = move.detail.replace(...)  # or rebuild from real numbers
```

**Severity:** high — the brief is the customer-facing distillation. If numbers are wrong, the entire "data-driven competitive intel" promise is broken. Even one obvious error damages trust.

**Quick mitigation today (no code change):** add a small disclaimer to the Brief tab — `"Move details are AI-generated; verify specifics against the Analytics tab."` It's not a real fix but it sets expectations honestly while we build the coercer.

## 4. What works well

Worth calling out — most of the pipeline IS solid:

- **Domain aggregation v1.1 fix** correctly excludes `no_data` floor scores. Pipeline doesn't hide problems behind fake zeros.
- **Category-aware WTP baselines** (Will's Day 1) — handbag pricing benchmarks instead of global average. Score makes sense for the category.
- **DeepSeek prompt quality** is high — verdict + moves are grounded in real brand names, not hallucinated competitors. Just the deltas need numerical-coherence checking.
- **Coercer drops bad brand IDs** — `古良吉吉` and `CASSILE` always appear correctly; never invents "Coach" or other untracked brands.
- **`run_analysis_for_workspace.sh` orchestrator** runs end-to-end in 12 seconds. That's fast.
- **`USE_MOCKS=false` doesn't crash.** Empty workspaces just show empty state cleanly.

## 6. The bigger picture

The platform is in an interesting state right now:

- **Architecture:** Solid — 4-layer separation (scrape → score → narrate → present) is clean, each layer testable in isolation
- **Pipelines:** Built and running — Will shipped the full 7-pipeline AI synthesis layer in 2 days
- **Frontend:** Renders real data — no mocks, empty states honest
- **Data quality:** **The weak link.** Both because scrapes are sparse + occasionally buggy, AND because the scoring layer is too forgiving of bad inputs

**The product is correctly designed but is being fed garbage in a few specific places.** Fix the data-quality issues and the same code will produce 5× sharper output.

This is a much better problem to have than the inverse (good data, bad pipelines).
